End-to-End Workflow
  1. Upload — drag & drop input files. SD: a single XML. MM: three xlsx files (Main, Alternate UoM, Long Text). The system detects file types automatically and runs structural checks before parsing.
  2. Parse + Merge — the loader reads all rows in seconds (Rust-backed calamine path), then the merger groups rows by MATNR. For MM, multi-plant materials are detected (same MATNR appearing at multiple WERKS) and tracked as a single material with a list of plant rows.
  3. Validate — every material is checked against the rule catalogue (47 MM rules, ~30 SD rules). Each failure produces an Error object tagged with the row, column, rule, severity, and a suggested fix where the rule can derive one.
  4. Group into Decisions — errors are clustered: when many rows have the same problem with the same value, they collapse into one "pattern" Decision. Otherwise the rule produces "individual" Decisions (one per affected row).
  5. Resolve — for each Decision, SMEs can either:
    — Group & Replace: pick from a KDS-backed dropdown (or type a value) to update every matching cell at once.
    — Fix Individually: edit each row by hand in the record editor.
  6. Export — for MM, two output options:
    — LTMC XML: directly upload-ready SpreadsheetML matching HML's LTMC template byte-for-byte. Use this for the actual SAP migration.
    — Cleaned xlsx: 5-sheet bundle (Summary / Main / AlternateUnits / LongText / ChangeLog) for review-before-upload.
MM Validation Rule Categories

47 rules across 8 categories. The validator iterates the catalogue once per material; rules that depend on missing KDS catalogs stay dormant rather than firing false positives.

Category Triggers when… Example rule
Mandatory A required SAP field is blank. mm_base_unit_missing (MEINS)
Max length A field exceeds its SAP-defined width. mm_description_too_long (MAKTX > 40)
Exact value A field doesn't match HML's required value (e.g. "must always be X"). mm_industry_sector_must_be_H (MBRSH)
KDS lookup The value isn't in the reference catalog (Plant, Material Group, EXTWG, EKGRP, etc.). mm_matgroup_not_in_kds (MATKL)
KDS nested lookup The value is valid in the catalog but not in the current scope (e.g. an LGORT not approved at this WERKS). mm_storage_loc_not_in_kds (LGORT × WERKS)
MATNR range Material number doesn't match the MTART's allowed range (numeric vs alphanumeric, length). mm_matnr_range_mismatch
Conditional mandatory Field A is required only when Field B has a specific value. mm_inhouse_prod_time_missing (DZEIT required when MTART ∈ in-house types)
LTMC default A field that LTMC requires but the source xlsx doesn't carry — system suggests the standard default. mm_ltmc_language_missing (SPRAS → EN)

Severity: error blocks LTMC migration — warning flagged for review but can ship as-is.

Multi-Plant Materials

The same MATNR can appear at multiple plants in the source file (e.g. one item stocked at PE01, SP01-SP09, and WH01 — 11 plant rows). The merger detects this and keeps a single material record with a list of plant rows.

Plant-scoped fields — values can differ per plant; the validator and export emit one row per plant for these:

WERKS · LGORT · LGPRO · DISPO · FEVOR · BWKEY

True duplicates — same MATNR and same WERKS appearing twice are detected and the second occurrence is dropped (with a warning shown on the dashboard). 1-MATNR-N-plants is normal; same-MATNR-same-WERKS is data entry error.

Group & Replace at scale — a single bulk fix for a plant-scoped field (e.g. clearing a bad LGPRO value) automatically reaches every affected plant row across every multi-plant material. On the Peenya FG file (10,565 materials × 11 plants = 24,575 plant rows), one G&R operation completes in ~3 seconds.

LTMC XML Export Structure

The exporter loads HML's filled-in LTMC template from disk and patches data rows into each sheet's data section. Everything else passes through unchanged — styles, freeze panes, column widths, worksheet protection, document properties, color formatting all preserved byte-for-byte.

Audit Trail

Every edit applied during a session is recorded in the audit log with: timestamp, username, action type (group_replace, edit_cell, etc.), rule, MATNR, WERKS, SAP field, old value, new value, business reason.

Cleaned xlsx export includes the full audit log as a "ChangeLog" sheet — every edit traceable to who made it, when, and why.

Bulk operations (Group & Replace, bulk delete) require a non-empty business reason before they run.
